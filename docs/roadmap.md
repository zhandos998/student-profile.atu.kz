# Student Profile Roadmap

Этот файл фиксирует текущую структуру и следующие шаги, чтобы не держать план только в переписке.

## Статус

### 1. Отдельная сущность групп

Статус: частично сделано.

Сделано:
- Создана таблица `student_groups`.
- `student_profiles` связан с группой через `student_group_id`.
- `group_social_passports` связан с группой через `student_group_id`.
- Студент выбирает группу из списка.
- Социальный паспорт показывает студентов выбранной группы.

Осталось:
- Создать отдельную таблицу `faculties`.
- Решить, оставляем имя таблицы `student_groups` или переименовываем в `groups`.
- Постепенно убрать зависимость от текстового `group_name`, сейчас он еще сохранен для совместимости и отображения.

### 2. Несколько групп у куратора/эдвайзера

Статус: сделано базово.

Сделано:
- Добавлена страница `/groups`.
- Можно создать несколько групп.
- У каждой группы свой социальный паспорт.
- Старый `/group-social-passport` редиректит на доступную группу.
- Добавлены фильтры на `/groups`: факультет, курс, куратор/эдвайзер.

Осталось:
- Добавить редактирование/архивацию группы.
- Уточнить, может ли эдвайзер владеть группой или только видеть/создавать портреты студентов.

### 3. Разделить анкету студента и служебные данные

Статус: сделано базово.

Сделано:
- Студент заполняет личные, учебно-организационные, контактные данные, проживание, социальный статус, социальную поддержку и академический профиль без аналитики.
- Добавлены поля `personal_email` и `parent_guardian_contacts`.
- Социальный блок имеет отдельный статус проверки: `social_review_status`, `social_review_comment`, `social_reviewed_at`, `social_reviewed_by_id`.
- Академический блок имеет отдельный статус проверки: `academic_review_status`, `academic_review_comment`, `academic_reviewed_at`, `academic_reviewed_by_id`.
- Эдвайзер/администрация могут отдельно подтвердить социальный и академический блоки или вернуть их на исправление с комментарием.
- При сохранении студентом социальный и академический блоки снова переходят в статус `pending`.
- Поля аналитики `grade_dynamics`, `group_comparison`, `success_forecast` пока не выводятся в анкете и не сохраняются через нее.

Осталось:
- Ограничить медицинские и психологические данные через permissions, а не только через текущие role-checks.

### 4. Статусы заполнения анкеты

Статус: сделано базово.

Сделано:
- Добавлены поля статуса в `student_profiles`: `profile_status`, `submitted_at`, `verified_at`, `reviewed_by_id`, `revision_comment`.
- При сохранении студентом анкета автоматически получает статус `submitted`.
- Эдвайзер/администрация могут отметить анкету как проверенную или вернуть на исправление с комментарием.
- В списке портретов есть фильтр и бейдж статуса анкеты.
- В карточке студента показывается текущий статус, даты отправки/проверки и комментарий к исправлению.

Статусы:
- `not_started` - не заполнена;
- `draft` - черновик;
- `submitted` - отправлена;
- `verified` - проверена куратором;
- `needs_revision` - требует исправления.

Осталось:
- Уточнить, кто именно имеет право финально проверять анкету: куратор, эдвайзер или администрация.
- Добавить историю изменений статусов в журнал действий.

### 5. Сервис рисков

Статус: сделано базово.

Сделано:
- Создан `StudentRiskService`.
- В сервис вынесены: заполненность анкеты, академические риски, социальные риски, психологические риски, медицинские риски.
- `DashboardController`, `AnalyticsDashboardController` и список портретов используют единую логику сервиса.
- Добавлен unit-тест на основные правила риска.

Сервис считает:
- академические риски;
- социальные риски;
- психологические риски;
- медицинские риски.

Осталось:
- Уточнить точные пороги риска по GPA и академической задолженности с заказчиком.
- Позже расширить сервис интеграционными данными из LMS, тестовой платформы и мед/психологических баз.

### 6. Permissions вместо только ролей

Статус: частично сделано на уровне методов в `User`.

Сделано:
- Добавлены единые методы доступа в `User`: кто видит все данные, кто управляет портретами, кто использует свою анкету.

Осталось:
- Создать таблицы permissions/role_permission.
- Добавить права:
  - `view_students`
  - `edit_student_profile`
  - `view_psychology`
  - `view_health`
  - `export_reports`
  - `manage_groups`
- Перевести проверки в контроллерах на permissions.

### 7. Журнал действий

Статус: не сделано.

Нужно логировать:
- кто изменил анкету;
- когда изменил;
- какие поля изменил;
- кто выгрузил отчет;
- кто смотрел/менял психологические и медицинские данные.

### 8. Главные страницы по ролям

Статус: частично сделано.

Сделано:
- У студента отдельный блок личной информации, успеваемости, достижений, портфолио и рекомендаций.
- У куратора/эдвайзера есть обзор студентов, рисков и аналитики.
- У администрации есть статистика, рейтинги, отчеты и мониторинг.

Осталось:
- Упростить студенту главную страницу до задач: заполненность, что обновить, достижения, портфолио, рекомендации.
- Для куратора вывести его группы, студентов без анкеты, риски и последние изменения.
- Для администрации добавить фильтры по факультету, курсу и группе.

## Ближайший порядок работ

1. Добавить журнал действий.
2. Перейти от ролей к permissions.
3. Ограничить медицинские и психологические данные через permissions.
4. Добавить редактирование и архивирование групп.
5. Уточнить правила финальной проверки анкеты.
